# Active Decisions

- **Architecture**: Adopted a Monorepo structure to simplify version control and deployment pipelines.
- **Backend Hosting**: Selected Railway's $5/mo plan to avoid cold starts and ensure high availability for the Python API.
- **Frontend Hosting**: Vercel selected for the Next.js dashboard due to optimized global edge performance.
- **Database Engine**: Transitioned to PostgreSQL on Railway to guarantee data persistence, replacing ephemeral SQLite.
- **Business Model / Pricing**: Adopted a 3-tier structure (Basic, Scale, Growth). Removed the free tier to monetize immediately.
- **Credential Architecture**: A single user account can now hold multiple distinct Google Ads credentials (Customer IDs), restricted by their subscription tier (1, 3, or Unlimited).
- **Python Constraints**: Pinned backend Python version to 3.11 for Railway deployments to retain the `crypt` module required by `passlib`.
- **Google Ads API Versioning**: `google-ads` library must be kept at v31+ to avoid `501 GRPC` errors from sunset API versions. Env vars for OAuth in `google_ads_service.py` must use `GOOGLE_ADS_CLIENT_ID` / `GOOGLE_ADS_CLIENT_SECRET` (matching `auth_google.py`).
- **Autopilot Architecture (Hybrid Model)**: Adopted a split execution model for the Google Ads orchestrator. Budget protection rules (PAUSE) are executed automatically via a background cron job. Scaling rules (SCALE) generate pending logs that require manual UI approval from the client. Additionally, a "Learning Phase" heuristic (minimum spend > 1.5x Target CPA) protects new campaigns from being shut down prematurely.
- **Domain Override**: Unconditionally force custom domain (googlemaker.thequantpartners.com) in CORS allowed origins to bypass Railway env variable overrides.
- **Agent Workflow**: External AI agents (e.g., Jules) do not push directly to `main`. They always submit their work via Pull Requests for safety and human review. Agents reviewing code must check remote branches.
- **Payment Gateway**: Integrated Lemon Squeezy as Merchant of Record (MoR) for subscription handling, replacing local mock tier assignments.
- **UI Architecture**: Selected Tailwind CSS v3 and Recharts for the primary frontend dashboard aesthetics, prioritizing a high-end dark mode, glassmorphism, and neon gradient design language over standard CSS modules. All new components must be mobile-responsive by default, utilizing sliding sidebars and hamburger menus for navigation.
- **Localization**: Client Dashboard and Landing Page default to English. Landing Page supports EN/ES toggle. SuperAdmin dashboard remains in Spanish. Plan benefits are positioned for Do-It-Yourself (DIY) automation rather than human-supported service.
- **Database Migrations & Production Env**: The local frontend environment (`.env.local`) points to the PRODUCTION API on Railway (`googlemaker-production.up.railway.app`), not `localhost:8000`. Any database schema changes must be accompanied by explicit migration logic (e.g., raw SQL in `lifespan` in `main.py`) and MUST be pushed to `main` so Railway deploys them. Otherwise, testing the local frontend will result in 500 API errors from the un-migrated production database.
- **Pixel / Campaign Mapping**: Instead of attempting to parse complex Account-Level versus Campaign-Level goals in the Google Ads API, the "Manage Pixels" feature determines which campaigns a pixel is anchored to by looking at historical data (`metrics.conversions > 0` over `LAST_30_DAYS` segmented by `conversion_action`). This provides marketers with practical "active" data rather than theoretical configuration.
- **Branding**: Dropped the 'Google' trademark from the product name to prevent legal, OAuth, and payment gateway issues. The brand is officially 'GMaker'.
- **Ideal Customer Profile (ICP)**: Product marketing and default templates are hyper-focused on **US Immigration Lawyers** (targeting the US Latino market).
- **Ad-Spend Based Pricing**: Pricing tiers are gated by managed Google Ads spend volume rather than feature access. All users get the full Autopilot core.
- **Soft Pause Policy**: When a user exceeds their tier's ad-spend limit, GMaker will **NEVER** pause their actual Google Ads campaigns. Instead, GMaker will execute a "Soft Pause", disabling its own Autopilot optimizations and displaying a warning to upgrade, protecting the client's underlying business operations from accidental disruption.
- **Legal & Compliance**: Overhauled all legal clauses (Terms, Privacy, Refunds) specifically tailored to the litigious US Immigration Lawyer market. Added strict liability caps, ad-spend waivers, indemnification, and mandatory arbitration clauses to protect the company.
- **AI Feature Data Strategy**: Adopted a manual "Saved Strategies Vault" approach instead of auto-saving all AI generation. This prevents database bloat from low-value generations and empowers the user to curate their own vault of high-value ad copy.
- **AI Content Extraction**: Instead of relying on user-provided descriptions, the AI strategy tool mandates a URL input to scrape the client's actual landing page for unique selling propositions, reducing user cognitive load and improving copy quality.
- **Notifications & Approvals (Telegram)**: Selected Telegram as the primary channel for Autopilot notifications over email. This enables real-time engagement and immediate action via Inline Keyboards without leaving the messaging app. Webhook handlers in the backend process these button clicks asynchronously to mutate Google Ads.
- **Branding Enforcement**: Project name must be strictly capitalized as GMAKER in all documentation, including README.md.
- **Strategic Focus (Feature Creep Prevention)**: Deliberately decided NOT to implement an Inbound SMS Acquisition Infrastructure (Twilio SMS Widget + Stripe checkout) for GMaker clients. The core value proposition remains strictly focused on the "Autopilot" loop: Google Ads Setup -> Campaign Generation -> Telegram Monitoring. Expanding into Lead Conversion at this stage would cause feature creep and dilute the product's primary focus.
- **Business Model Pivot (Growth Partner)**: Transitioned from selling GMaker as a standalone SaaS to offering a complete "Customer Acquisition System" (Web Development + Chat Widget + Google Ads Management via GMaker). GMaker serves as the operational leverage tool for "The Quant Partners".
- **Lead Conversion Expansion**: Reversed the previous decision to avoid lead conversion. We are now building a Chat Widget into GMaker that uses heuristic rules for pre-qualification and AI for closing, enabling the 360-service offer.
- **Chat Widget API Security (Sub-app Routing)**: To allow open CORS for the embeddable `gmaker-widget.js` script while protecting the private dashboard API, the widget endpoints are mounted as an isolated FastAPI sub-app (`/widget`) with `allow_origins=["*"]`.
- **AI Lead Extraction Pattern**: Instead of using complex Function Calling, the `chat_engine.py` instructs Gemini to emit a specific token string (`[[LEAD:{...}]]`) at the end of its response. The backend extracts this JSON using Regex to save the Lead into PostgreSQL. This improves reliability and speed.
- **Sub-App CORS Routing (FastAPI)**: To bypass the restrictive global `CORSMiddleware` (which protects the dashboard API but incorrectly intercepts all sub-app `OPTIONS` requests), a custom `WidgetCORSMiddleware` was introduced. It intercepts `/widget/` routes directly and injects `Access-Control-Allow-Origin: *`, ensuring the public widget can be embedded anywhere without compromising the main API's security.
- **BYOK (Bring Your Own Key) Enforcement**: To scale the AI Chat Widget without incurring massive LLM API costs or hitting global rate limits, we enforce a strict BYOK policy. Clients must supply their own API keys (OpenAI, Anthropic, or Gemini). Keys are AES-256 encrypted at rest in PostgreSQL and are never returned in plaintext to the frontend.
- **2026-06-29 - Manychat WhatsApp-First Integration:** Decided to use QSS backend as the LLM Brain and process webhook conversions using an Upsert logic to handle leads bypassing web forms.
- **2026-07-01 - YCloud WhatsApp Integration Migration:** Decided to migrate from Manychat to YCloud (direct WhatsApp Business API wrapper). GCLID parsing is now executed dynamically via regex on standard WhatsApp text messages (e.g. 'Ref: GCLID--...'). YCloud payload dynamic routing is maintained by appending '?client_id=' to the webhook URLs in the frontend dashboard.
- **2026-07-01 - YCloud Native Webhook Security:** Transitioned the YCloud architecture to use a single "Master Webhook" rather than separate chat and conversion endpoints. Security is strictly enforced using cryptographic HMAC-SHA256 signature verification matching YCloud's native Webhook Secret mechanism (`X-YCloud-Signature`).
- **2026-07-01 - WhatsApp Automation Strategy:** Decided to use unofficial libraries (like Baileys) strictly for internal testing and lead demonstration purposes, while reserving official Cloud APIs (YCloud) exclusively for client deployments to protect their Meta Business Managers from ban risks.
- **2026-07-01 - WhatsApp Anti-Ban (Baileys):** To prevent instant bans on unofficial WhatsApp bots, interactive buttons MUST be avoided (use text-based numbered lists instead). Responses must mathematically mimic human typing speed (minimum 4 seconds + length multiplier), and the bot must explicitly broadcast an `available` presence state before broadcasting `composing` to ensure the WhatsApp app actually renders the "typing..." indicator.
- **2026-07-01 - Railway Config-as-Code:** When using `railway.toml`, the builder for Docker must be explicitly defined as `builder = "DOCKERFILE"` (not `DOCKER`). Incorrect syntax will result in silent deployment rejections where Railway ignores webhooks and continues running old containers, masking underlying code updates. Additionally, defining `[volumes]` in `railway.toml` will create and mount a new empty volume, replacing any manually mounted volume in the UI.
- **2026-07-02 - Railway Volume Management:** Railway's new Canvas UI ignores the [volumes] section in ailway.toml. Volumes must be created manually as standalone nodes on the Canvas and attached to the service via Mount Paths (e.g., /app/auth_info_baileys) to ensure session persistence across container restarts.
- **2026-07-02 - Branding Update:** Officially rebranded the platform from GMaker to QSS (Quant System Sales).
- **2026-07-03 - OpenWA vs Baileys Architecture:** Evaluated integrating the OpenWA Gateway directly into the QSS backend as an alternative to the standalone Baileys server. The OpenWA integration was implemented and subsequently reverted. The decision was made to strictly maintain the lightweight, already-working Baileys architecture for end-clients, preventing infrastructure bloat and complex internal Docker networking issues on Railway.
- **2026-07-03 - Advanced WhatsApp Anti-Ban (Business Hours):** Meta's detection algorithms target 24/7 instant response bots. To mitigate this, a Business Hours scheduling feature was implemented. Crucially, the schedule logic executes *after* the rule-based qualification phase (`RULES_MODE`) but *before* the LLM handoff (`AI_MODE`). This ensures the bot safely operates 24/7 to capture leads, but suppresses the obvious bot-like behavior of holding long generative conversations at 4 AM, perfectly mimicking official WhatsApp Business 'Away Message' functionality.

- **Cal.com Function Calling:** Implemented native tool-calling loops across 3 providers (Gemini, Anthropic, OpenAI) to execute get_availability and book_meeting instead of relying on LangChain, keeping the backend lightweight.
